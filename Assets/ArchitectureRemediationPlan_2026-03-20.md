# 项目框架设计整改计划

- 生成时间: 2026-03-20 18:26:07 +08:00
- 项目名称: Mahjong
- 文档用途: 汇报版整改方案
- 依据来源: 本次对项目框架设计的静态代码审查结果

## 一、整改背景

当前项目已经具备一定的框架雏形，主要包括:

- 游戏状态机
- 事件总线
- UI 基类体系
- 业务服务层
- 场景切换工具层

从开发方向看，项目具备较好的结构意识；但从工程稳定性和后续扩展性看，目前仍存在以下核心问题:

- 状态机主链路未完全闭合
- 状态迁移约束没有真正生效
- 全局事件监听生命周期管理不统一
- 状态副作用分散在多个模块中
- 领域模型与 Unity 场景对象耦合较深
- 业务层尚未形成编译期边界

这些问题在单机原型阶段还能勉强承受，但如果后续要支持联网、AI 托管、回放、更多玩法扩展或多人协作开发，就会逐渐成为系统稳定性和演进效率的主要瓶颈。

## 二、整改目标

本次整改的总体目标不是“推翻重写”，而是在保留当前项目主体结构的前提下，完成从“目录分层”到“真实架构约束”的升级。

本次整改建议聚焦四个目标:

### 目标 1: 补齐运行时主链路

保证游戏从大厅、选庄、发牌、对局到结算的状态流转完整、可控、可验证。

### 目标 2: 收紧框架约束

把目前依赖人工约定的规则，收口到统一的状态机入口、事件注册模板和生命周期机制中。

### 目标 3: 推进领域层解耦

让核心麻将规则、玩家数据、流程编排逐步脱离 Unity `MonoBehaviour`，为网络同步、测试和 AI 扩展打基础。

### 目标 4: 为后续长期演进建立边界

通过程序集划分和职责重整，降低后续功能叠加对现有代码的破坏性。

## 三、核心问题与整改方向

### 1. 状态机主链路不完整

现状:

- `GameState` 已定义 `BankerSelection`
- UI 启动逻辑也会尝试进入该状态
- 但 `GameFlowController` 中未完成该状态注册

风险:

- 启动流程会在关键入口中断
- 状态枚举与状态实现不一致
- 后续新增流程节点仍可能重复出现类似问题

整改方向:

- 统一状态注册入口
- 将状态枚举、状态实例表、跳转图维护在同一处
- 为缺失状态注册增加启动期校验

### 2. 状态迁移机制缺乏硬约束

现状:

- `CanTransitionTo()` 已存在
- 但非法迁移时仍继续执行状态切换

风险:

- 状态机约束失效
- 任意模块可能绕过流程推进系统
- 缺陷不容易在早期暴露

整改方向:

- 非法迁移立即中断
- 所有状态迁移统一通过单入口执行
- 建立最小化状态切换日志

### 3. 事件系统生命周期管理不统一

现状:

- 事件系统已经提供自动解绑能力
- 但部分请求处理器未接入这套机制

风险:

- 场景切换后重复监听
- 悬空委托导致重复处理
- 全局事件总线逐步失控

整改方向:

- 为 `BaseUIRequestHandler` 增加统一的注册/销毁模板
- 所有监听必须使用自动解绑或集中回收列表
- 区分全局事件与场景级事件

### 4. 状态对象没有完整拥有自己的副作用

现状:

- 某些状态类只承担“名义状态”职责
- 真正的流程推进、事件绑定、完成条件散落在 Service、Manager、UI 中

风险:

- 流程难以维护
- 状态边界模糊
- 新增玩法时容易继续扩散耦合

整改方向:

- 每个状态对象完整拥有自己的进入、执行、完成、退出逻辑
- 将完成条件、事件绑定和下一个状态决策收口到状态内部
- Service 仅保留纯业务能力，不直接承担流程主导职责

### 5. 领域模型与 Unity 表现层耦合过深

现状:

- `PlayerModel` 直接保存 `Player` 组件
- Service 层继续围绕 Unity 对象传递和处理

风险:

- 不利于联网同步
- 不利于单元测试
- 不利于回放、服务器权威逻辑和 AI 推演

整改方向:

- 引入纯数据玩家模型
- `Player` 组件只作为 View/Adapter
- Domain 层只处理数据和规则，不处理 Unity 生命周期

## 四、分阶段整改计划

## 第一阶段: 稳定当前框架骨架

建议周期: 1 到 3 天

目标:

- 把当前容易中断流程的问题先修平
- 先保障项目能稳定推进开发

整改项:

- 补齐 `BankerSelectionState` 注册
- 修复非法状态迁移仍继续执行的问题
- 修复状态进入/退出事件绑定方向错误
- 统一 `LobbyUIRequestHandler` 等对象的事件解绑方式
- 补充关键流程日志

阶段产出:

- 状态机主流程可跑通
- 事件系统不会因场景切换产生明显重复监听
- 核心状态切换具备可追踪性

## 第二阶段: 收敛职责边界

建议周期: 3 到 5 天

目标:

- 让“状态机负责流程，Service 负责业务，UI 负责表现”这件事真正落地

整改项:

- 梳理 `GameFlowController` 与各状态类职责
- 将状态完成条件统一回收到状态对象
- 减少 Manager 对流程细节的直接控制
- 让 UI Request Handler 仅负责转发用户意图，不直接承载业务推进

阶段产出:

- 状态对象成为真正的流程节点
- Service 层更偏纯业务计算
- 流程结构更清晰，便于扩展新状态

## 第三阶段: 领域层解耦

建议周期: 5 到 10 天

目标:

- 为联网、测试、AI 和回放能力打基础

整改项:

- 定义纯数据 `PlayerData`、`RoomData`、`RoundData`
- 将 `PlayerModel` 从 `Player` 组件中脱离
- `WinChecker`、`ScoreCalculator`、牌局规则逐步只依赖纯数据对象
- 保留 `Player` MonoBehaviour 作为场景表现层适配器

阶段产出:

- 规则层脱离 Unity 生命周期
- 核心算法可被单元测试直接覆盖
- 为 Mirror 接入建立更合理的数据边界

## 第四阶段: 建立编译期边界

建议周期: 2 到 4 天

目标:

- 把“约定式分层”升级为“程序集级分层”

整改项:

- 引入业务 asmdef
- 建议拆分为:
- `Mahjong.Domain`
- `Mahjong.Application`
- `Mahjong.Presentation`
- `Mahjong.Infrastructure.Unity`

说明:

- `Domain` 放规则、模型、算法
- `Application` 放流程编排、服务、用例
- `Presentation` 放 UI、页面、面板、视图
- `Infrastructure.Unity` 放 MonoBehaviour、Scene、Unity 适配器

阶段产出:

- 跨层依赖在编译期就能暴露
- 团队协作边界更清晰
- 后续模块增长时维护成本更可控

## 五、建议优先级

### P0: 必须立即处理

- 状态注册缺失
- 非法状态迁移未拦截
- 事件监听未统一解绑
- 选庄状态流程闭环不完整

### P1: 应在近期处理

- 状态职责与服务职责重新划分
- 流程编排从 Manager 中下沉到状态/应用层
- 场景切换纳入统一流程管理

### P2: 中期持续推进

- 纯数据领域层拆分
- Mirror 接入边界规划
- asmdef 模块化

## 六、预期收益

完成上述整改后，项目将获得以下收益:

- 游戏主流程更稳定，运行时问题更容易定位
- 状态机真正成为系统流程中心
- 事件系统更可控，场景切换更安全
- 业务规则更容易测试和复用
- 为联网、AI、回放等能力预留清晰扩展位
- 后续多人协作开发的冲突成本降低

## 七、实施建议

为了降低风险，建议采用“边修边验证”的方式推进，而不是一次性大重构。

建议顺序:

1. 先修复状态机和事件系统中的确定性缺陷
2. 再整理状态职责和流程编排
3. 然后推动领域层数据化
4. 最后进行程序集拆分

这样可以避免因为一次性拆分过多而影响当前开发节奏。

## 八、汇报结论

该项目当前具备良好的框架演进基础，但还处在“结构初具雏形、机制尚未完全闭环”的阶段。

如果项目目标仅为课程原型或短期单机演示，当前结构在修复关键问题后可以继续使用；但如果项目目标包含长期迭代、联网同步、多人房间、AI 托管、规则扩展或工程化协作，那么建议尽快按照本整改计划推进。

总体判断如下:

- 当前架构方向正确
- 当前运行时约束不足
- 当前领域层解耦不够
- 当前非常适合进行一次中等规模、分阶段的结构整改

## 九、备注

本整改计划基于当前代码静态审查结果整理，未结合完整运行时联调结果。
