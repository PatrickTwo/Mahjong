# 当前分层结构

本文档用于说明项目当前已经落地的分层方式，帮助后续继续迭代时保持职责清晰。这里不追求理论完整，重点是方便开发时快速判断代码应该放在哪一层。

## 一、当前分层

### 1. UI 视图层
- 目录示例：`Assets/Game/Scripts/Core/UI/`
- 典型对象：`Page`、`Panel`、`View`、`Widget`
- 当前代表：
  - `LobbyPageUI`
  - `JoinRoomPanelUI`
  - `PlayerListViewUI`
  - `PlayerCardUI`
  - `PlayerOperationPanelUI`

职责：
- 负责界面显示、按钮绑定、面板开关、基础交互
- 把用户输入转给 Presenter
- 根据 ReadModel 刷新界面

不负责：
- 不直接写大厅业务规则
- 不直接推进游戏流程
- 不直接操作底层 Manager 的复杂逻辑

### 2. Presenter 层
- 目录示例：
  - `Assets/Game/Scripts/Core/UI/Lobby/Presenter/`
  - `Assets/Game/Scripts/Core/UI/Game/Presenter/`
- 当前代表：
  - `LobbyPresenter`
  - `LobbyReadModel`
  - `GameHudPresenter`
  - `GameHudReadModel`

职责：
- 承接 UI 输入
- 编排页面级逻辑
- 调用 Service
- 维护 UI 所需的只读状态

说明：
- `LobbyPresenter` 负责大厅主页面交互
- `GameHudPresenter` 负责庄家选择和对局 HUD 的展示状态收口
- `ReadModel` 只给 UI 读，不给 UI 直接改

### 3. Application Service 层
- 目录示例：`Assets/Game/Scripts/Core/Application/`
- 当前代表：
  - `LobbyService`
  - `LobbyAudioSettingService`
  - `UIFlowService`
  - `EventBusService`

职责：
- 处理跨对象的业务编排
- 对 Presenter 暴露稳定接口
- 收口事件系统、页面切换、大厅业务和音频设置等能力

说明：
- `UIFlowService` 统一处理界面打开和关闭
- `LobbyService` 负责大厅相关业务动作
- `EventBusService` 作为事件总线适配层，减少业务代码直接依赖 `EventSystemManager.Instance`

### 4. 流程与领域层
- 目录示例：
  - `Assets/Game/Scripts/Core/GameControl/`
  - `Assets/Game/Scripts/Core/GameControl/States/`
- 当前代表：
  - `GameFlowController`
  - `BankerSelectionState`

职责：
- 负责游戏状态切换
- 负责流程推进
- 负责领域事件发出

不负责：
- 不决定某个按钮怎么显示
- 不直接操作具体 UI 组件

### 5. Manager 层
- 目录示例：`Assets/Game/Scripts/Core/Manager/`
- 当前代表：
  - `MahjongGameManager`
  - `PlayerManager`

职责：
- 持有运行期对象和全局协调能力
- 为流程层和服务层提供基础能力

说明：
- 当前项目里 Manager 仍然比较重要
- 后续如果继续演进，建议逐步把业务判断从 Manager 下沉到 Service 或更纯粹的领域对象中

## 二、当前推荐调用链

### 大厅链路
`UI(View) -> LobbyPresenter -> LobbyService / UIFlowService -> Manager / Flow -> LobbyReadModel -> UI`

### 对局 HUD 链路
`UI(View) -> GameHudPresenter -> EventBusService / Flow -> GameHudReadModel -> UI`

## 三、当前约束

- UI 层尽量只依赖 Presenter 和 ReadModel
- Presenter 不直接写复杂规则，复杂业务交给 Service
- 游戏流程统一由 `GameFlowController` 和状态对象推进
- 跨模块事件统一优先通过 `EventBusService` 收口
- 新功能优先沿用现有分层，不再增加新的 `UIRequestHandler`

## 四、后续优化方向

- 继续把对局 UI 按 `Presenter + ReadModel` 方式补齐
- 逐步减少 Manager 中的业务判断
- 逐步把单例依赖改成更清晰的注入或装配方式

## 五、一句话原则

UI 只负责显示和输入，Presenter 负责页面编排，Service 负责业务收口，Flow 负责流程推进，ReadModel 负责给 UI 读数据。
