## Mahjong 项目框架设计

### 一、设计目标
- **可扩展**：规则、流程、UI、联网/单机、AI 等能力可逐步加入而不推翻架构。
- **可维护**：核心规则与表现层解耦，减少“到处拿单例改状态”的隐式依赖。
- **可测试**：核心逻辑尽量不依赖 Unity 场景对象，可做纯逻辑验证与回放。
- **可迭代**：优先保证最小闭环可运行，后续以模块方式替换/增强。

### 二、总体原则（依赖方向）
- **核心领域（规则/流程）不依赖 UI**；UI 仅订阅状态变化并发起“用户意图”请求。
- **单向数据流**：输入（命令/请求）→ 业务执行（用例）→ 状态更新（Model/Store）→ 通知（事件）→ UI 渲染。
- **显式通信**：跨层只通过“请求/事件/只读查询”三种方式，避免相互直接调用内部对象。
- **组合优于继承**：规则与计分使用可插拔策略/配置，而非堆叠复杂继承树。

### 三、分层结构（建议目录与职责）
> 目标是“按职责分层”，而不是按功能杂糅在一个 Manager 里。

#### 3.1 表现层 Presentation（Unity UI / Scene）
- **包含**：Page、Panel、Widget、动画、输入交互、音效触发（可选）。
- **职责**：
  - 显示/隐藏 UI，绑定按钮与交互。
  - 订阅“UI 需要关心的事件/状态”，将状态渲染为界面。
  - 通过“请求入口”提交用户意图，不直接改核心状态。
- **禁止**：
  - 直接操作牌池、玩家手牌等核心对象；
  - 在 UI 内实现规则判断（例如可碰可杠可胡）。

#### 3.2 UI 导航层 UI Navigation（UIManager）
- **包含**：UIManager、页面栈/路由、Panel 分层（HUD/Popup/Toast）、遮罩/输入屏蔽、弹窗队列（可选）。
- **职责**：
  - 管理 Page 的打开/关闭（推荐栈：Push/Pop/Replace）。
  - 管理 Panel 的显示隐藏与层级、互斥关系。
  - 负责 UI 生命周期（创建、显示、隐藏、销毁）与资源加载策略（可选）。
- **不做**：任何麻将业务逻辑与网络协议。

#### 3.3 应用层 Application（用例 / 请求处理）
- **包含**：UIRequestHandler（或命名为 GameUseCase / LobbyUseCase）、输入命令定义、用例编排。
- **职责**：
  - 接收 UI 的“用户意图”（例如：开始游戏、出牌、声明听牌、准备、重开）。
  - 调用领域层执行，并将结果写入状态仓库（或触发领域事件）。
  - 处理跨领域编排（例如：状态切换 + 发事件 + 记录回放）。
- **特点**：对上稳定（UI 调用不频繁变化），对下可替换（规则/网络/AI 可替换）。

#### 3.4 领域层 Domain（规则与流程核心）
- **包含**：GameFlow（状态机/回合流程）、RuleEngine（规则集合）、WinChecker（胡牌判定）、ScoreCalculator（计分）、Player/Hand/Tile 等核心模型。
- **职责**：
  - 描述麻将的规则、回合推进、状态变更合法性校验。
  - 产出“领域事件”与“状态变更”，不关心 UI/动画/表现。
- **建议**：
  - 流程使用“有限状态机（FSM）/状态模式”组织；
  - 规则、计分采用“策略集合 + 配置”组织（便于换玩法）。

#### 3.5 基础设施层 Infrastructure（可替换实现）
- **包含**：存档、回放、日志、联网、AI、资源加载、配置系统等。
- **职责**：
  - 提供领域层/应用层所需的外部能力，但通过接口或门面隔离实现细节。

### 四、核心模块划分（建议）
- **Game Runtime（游戏运行时）**：承载对局的整体生命周期（创建局、开始、结束、重置）。
- **Flow / Turn System（流程与回合）**：状态切换、轮到谁、可行动窗口、优先级仲裁。
- **Rules（规则系统）**：可做一组规则组件，按顺序评估与裁定（可胡/可杠/可碰/可听等）。
- **Hand & Tile（牌与手牌）**：牌池、抓牌/弃牌、明刻/暗刻/杠、弃牌池等数据结构。
- **Win & Ting（听牌/胡牌）**：听牌集合计算、胡牌判定、特殊牌型扩展点。
- **Scoring（计分）**：番型/倍率/结算分配策略，可配置。
- **Eventing（事件系统）**：统一一套事件机制用于跨模块通知（见第六节）。
- **UI（大厅/对局/结算）**：按 Page/Panel 组合，逻辑通过请求入口驱动。
- **Persistence & Replay（存档/回放）**（可选）：记录输入命令与随机种子，支持复盘与调试。
- **Networking（联网）**（可选）：命令同步、房间/准备/断线重连，尽量复用应用层命令与状态快照。

### 五、分层通信模式（只允许三类）

#### 5.1 请求（Command / Intent）
- **方向**：UI → 应用层（请求入口）→ 领域层
- **语义**：表达“玩家想做什么”，而非“UI 想把数据改成什么”。
- **原则**：
  - 请求应可记录（用于回放/联网）；
  - 请求的处理结果由领域层决定合法性与状态变更。

#### 5.2 事件（Event）
- **方向**：领域层/应用层 →（发布事件）→ UI/其他订阅者
- **语义**：告知“发生了什么”（状态变化、行动结果、回合推进）。
- **建议**：
  - 事件载荷使用“不可变快照/DTO/只读数据”，避免 UI 持有可变引用误改状态；
  - 事件命名聚焦业务（例如“回合开始”“可行动集合变化”“对局结束”），避免 UI 事件与领域事件混用。

#### 5.3 查询（Query / Read Model）
- **方向**：UI → 只读状态（Store/Model）
- **语义**：UI 拉取渲染所需数据（手牌列表、当前玩家、可用操作集合等）。
- **建议**：
  - UI 不直接访问领域对象的可写接口；
  - 复杂 UI 组合可使用“只读 ViewModel/Presenter 输出”降低 UI 逻辑复杂度。

### 六、事件系统与生命周期（统一策略）
- **统一事件机制**：项目中应只保留一套跨模块事件系统，避免团队使用混乱。
- **订阅生命周期**：
  - UI 订阅应与 Page/Panel 生命周期绑定（显示时订阅、隐藏/销毁时解绑）。
  - 若使用“自动解绑”，必须确保实现是类型安全且无反射崩溃风险，并且不会在高频路径造成性能问题。
- **线程约束**：约定所有事件发布与 UI 更新在主线程执行（Unity 友好）。

### 七、UI 架构（Page / Panel / Widget）
#### 7.1 Page（页面）
- **定义**：代表一个主要业务场景（大厅、对局、结算、设置）。
- **管理**：由 UIManager 以“导航栈/路由”方式管理，Page 间尽量互斥。

#### 7.2 Panel（面板）
- **定义**：页面内可组合的功能块（信息栏、操作栏、聊天、弹窗、提示）。
- **管理**：由 UIManager 按层管理（HUD/Popup/Toast），支持并存与互斥规则。

#### 7.3 UI 与逻辑的连接方式（推荐）
- UI 只依赖“请求入口（用例层）”与“只读状态/事件”，不依赖具体领域对象。
- UIManager 只管展示与导航，不掺杂对局规则与流程控制。

### 八、开发与迭代思路（推荐路线）
- **阶段1：最小可玩闭环**
  - 建立：对局生命周期、回合推进、摸打流程、基础胡牌判定、基础结算、基础 UI。
  - 目标：可稳定跑完一局，事件与请求链路打通。
- **阶段2：规则与表现增强**
  - 听牌、碰杠优先级仲裁、特殊牌型、动画与操作提示、弹窗队列。
- **阶段3：可扩展能力**
  - 回放/复盘、AI、联网与断线重连、配置化玩法（多套规则/计分）。

### 九、扩展点清单（未来需求不推翻架构）
- **换玩法**：替换/组合 RuleEngine 与 ScoreCalculator 的策略集。
- **加联网**：复用“请求（命令）”作为同步单位；状态用快照/增量事件同步。
- **加回放**：记录“命令序列 + 随机种子 + 版本号”，可复现一局并用于调试。
- **加观战/录像**：同样基于命令与状态快照。
- **加多 UI 主题**：UI 与领域解耦后，只需替换表现层资源与渲染逻辑。

### 十、架构验收标准（健康度）
- **移除任意 UI Page** 不应导致核心逻辑无法运行（最多缺显示）。
- **核心领域目录** 不应引用任何具体 UI 类或 UI 资源。
- **UI 层** 不直接改核心数据结构；所有业务动作必须经由请求入口。
